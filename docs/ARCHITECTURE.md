# ARCHITECTURE — R1Fрейтинг

Описание продукта и устройства приложения. Простым языком — для продакта.

## Для кого это

**R1Fmabes** — стример на YouTube / Twitch / Boosty. Чем занят:

- Баттл-рэпер (сам выступает + смотрит чужие баттлы, делает реакции).
- Реактор: слушает альбомы и треки, разбирает их на стримах.
- Смотрит фильмы и аниме «по донатам» от зрителей.
- Пишет свои треки.

Сайт нужен трём аудиториям:

1. **R1Fmabes** — быстро заводит карточку «что слушал / смотрел / читал», ставит оценку, делится зрителям одной ссылкой вместо «найди в стриме на 2:14:30».
2. **Зрители** — заходят и видят всё в одном месте: что R1Fmabes послушал, как оценил, на какие баттлы делал реакцию.
3. **Кирилл (админ)** — пока сам заводит и правит контент от имени R1Fmabes. Когда продукт устаканится, передаст ввод самому R1Fmabes.

## Что делает сайт сейчас

Сайт-каталог авторских оценок с зрительским голосованием. Три типа записей:

- **Альбом** — список треков с оценкой каждого, итог считается как среднее (`scoreMode: 'auto'`). Можно вручную переопределить. Зритель может оценить альбом целиком или каждый трек отдельно. Стример и зритель отмечают «лучшие треки» (звёздочка, до 3).
- **Баттл** — стороны A и B (по одному бойцу или команда), любое количество раундов, у каждого раунда свой победитель (A / B / ничья), плюс итоговый победитель. Зритель голосует за победителя баттла + ставит цифровую оценку 0..10. *В планах третий формат — дезматч: N участников все против всех (см. [BACKLOG.md](BACKLOG.md) → «Дезматч-баттлы»).*
- **Отдельный трек** — одна ручная оценка (`scoreMode: 'manual'`), без треклиста. Зритель ставит свою цифровую.

К любой записи можно прицепить:

- Обложку, год, жанр, описание, ревью в Markdown.
- Ссылки на оригинал (Яндекс.Музыка, Spotify, Apple Music, YouTube Music, VK, SoundCloud, Bandcamp).
- Ссылки на реакции — **это видео самого R1Fmabes**, где он обозревает запись (YouTube, Boosty, Twitch, VK Видео, Rutube).

**На каждой карточке две капсулы оценок**: R1F (`item.finalScore`) и средняя зрителей (агрегат `viewerVotesByItem` из Store). В hero страницы записи — те же две оценки крупно, со счётчиком «N голосов». В треклисте альбома — три колонки «R1F · ТЫ · ВСЕ».

Главная: featured-полоса сверху (по лучшей карточке каждого типа), ниже поиск, сортировка по новым / лучшим / худшим, фильтр периода (`за всё время`, `за месяц`, `за 7 дней`, `в этом году`, `за прошлый год`) и переключатель базы даты — **оценённые** или **вышедшие**. Фильтры кешируются в `cachedHomeFilters` — переживают навигацию (открыл карточку → назад → фильтр остался), F5 сбрасывает.

В шапке: бегущая строка-анонс концерта (авто-исчезает по дате), кнопка «Войти» для зрителей (через Google OAuth), у админа — пункт «Админка». Бейдж 16+ и ссылки на политику/условия в футере.

## Куда продукт движется (видение)

В порядке приоритета:

1. **Расширение типов контента.** Сейчас альбом / баттл / трек. В планах:
   - **Онлайн-баттл** — отдельный тип, два мини-альбома противопоставлены друг другу (без раундов).
   - **Дезматч-баттл** — N участников все против всех, побеждает один.
   - **Фильм** — название, год, режиссёр, обложка, оценка, ссылка на источник.
   - **Аниме** — то же + кол-во серий / сезон.
2. **Расширенные фильтры** — по площадке (Versus / SLOVO / Кубок МЦ), по сезону, по этапу, по жанру, по «есть ли реакция R1Fmabes», сортировка по оценке зрителей.
3. **Импорт с других платформ** — Apple Music / Spotify / Bandcamp для треков, Kinopoisk / Shikimori для фильмов и аниме.
4. **Удаление аккаунта зрителем** — по 152-ФЗ право субъекта.

- **Анонимный посетитель** — видит весь опубликованный каталог, страницы записей, оценки R1Fmabes и средние зрителей. Голосовать не может (нужен логин).
- **Зритель с Google-аккаунтом** — входит через OAuth, при первом входе подтверждает условия (`viewer_profiles.consented_at`). Может ставить оценку трекам/альбомам/баттлам, выбирать победителя баттла, отмечать свои «лучшие треки». Один голос на сущность.
- **R1Fmabes** — два аккаунта в `admin_users`: настоящий Google `r1fmabes@gmail.com` (с 2026-05-28) и старый email+password `r1fmabes.rating@gmail.com`. Ведёт каталог через админку.
- **Админ** (Кирилл) — `kirillmakarov820@gmail.com`. Заводит и редактирует записи. Права через таблицу `admin_users` (см. [db/migrations/001_init.sql](../db/migrations/001_init.sql)). Кнопка «Админка» в шапке появляется по `is_admin()` из RLS.

## Хранение данных

Прод-БД живёт в Supabase Cloud, проект `nfekasqbzwjelrwyxqmv` (KirillMaller's Org). Регион Ireland, Free Tier. Миграции 001-011 накачены через Management API.

Два режима работы клиента, выбираются автоматически по наличию env-переменных:

### Демо-режим (без `.env.local`)

Всё в `localStorage` под ключом `r1frating-items-v1`. Авторизация — фейковая, флаг `r1frating-demo-admin`. Голосование зрителей в демо-режиме не работает. Удобно для разработки без интернета.

### Supabase-режим (есть `.env.local`)

`VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY` → `createClient`. Авторизация через Supabase Auth — email+password для админов (через `/admin/login`) и Google OAuth для всех (через кнопку «Войти» в шапке). RLS отделяет публичные чтения от админских и защищает голоса зрителей.

## Схема БД

Полный SQL — [db/migrations/001_init.sql](../db/migrations/001_init.sql). Кратко:

- **`rated_items`** — основная таблица записей. `type`, `slug` (уникальный), `final_score`, `score_mode`, `published`, `release_year`, `reviewed_at`, `metadata jsonb` (для баттлов и Яндекс-источника).
- **`track_scores`** — треки с оценкой, привязаны к `rated_items` через `item_id`. Уникальная позиция в рамках одного item. Колонка `is_best boolean` — стример отмечает «лучшие треки альбома».
- **`media_links`** — ссылки на оригиналы и реакции. `kind in ('original', 'reaction')`.
- **`admin_users`** — кто админ. `user_id` ссылается на `auth.users`.
- **`auction_items`**, **`auction_rules`** — аукционы (миграции 003, 004).
- **`viewer_votes`** — голоса зрителей (с миграции 007 + расширения 009/010/011). Колонки: `viewer_id`, `item_id`, `score numeric(4,2)`, `round_index int`, `track_position int`, `winner_side text`, `is_best boolean`. Семантика голоса определяется комбинацией:
  - `track_position = null, round_index = null, score != null` → оценка альбома целиком (или одиночного трека, или цифровая оценка баттла).
  - `track_position = N, round_index = null, score != null` → оценка трека N в альбоме.
  - `track_position = null, round_index = K, winner_side != null` → голос за победителя раунда K баттла.
  - `track_position = null, round_index = null, winner_side != null` → голос за итог баттла.
  - `is_best = true` совмещается с любой score-записью трека.
  - Score и winner_side **независимы в одной строке** — `saveMyAlbumVote` и `saveMyBattleFinalVote` пишут в одну строку (`track_position = null, round_index = null`) разные колонки, не затирая соседнюю.
  Уникальный индекс `(viewer_id, item_id, coalesce(round_index, -1), coalesce(track_position, -1))` — один зритель = один голос на (item, round, track-position). RLS: select всем для агрегации, insert/update — только своему viewer_id и только при `consented_at != null` в `viewer_profiles`.
- **`viewer_profiles`** — согласие зрителя на обработку ПДн (миграция 008). `user_id pk`, `consented_at`, `consent_version`. RLS «свой профиль только сам». Без записи здесь голосование заблокировано на уровне БД (миграция 009).

### RLS правила

- Все могут читать опубликованные `rated_items` и их треки + ссылки.
- Все могут читать `viewer_votes` для агрегации, но `viewer_id` (Google UUID) при этом виден — privacy-вопрос отложен до публичных профилей (см. backlog P3).
- Писать в `rated_items` / `track_scores` / `media_links` / аукционы — только админ (`is_admin()` функция).
- Голосовать (`viewer_votes`) — только аутентифицированный зритель с `consented_at` в своём `viewer_profiles`. Может писать только свой голос.
- Список админов виден только админам.

## Доменные типы (frontend)

Источник — [src/main.tsx](../src/main.tsx). Главное:

- `ItemType = 'album' | 'battle' | 'track'`
- `ScoreMode = 'auto' | 'manual'` — авто = среднее по трекам, manual = руками.
- `BattleSideKey = 'a' | 'b' | 'draw'` (legacy) и `BattleSide = 'a' | 'b' | 'draw'` (для зрительских голосов) — победитель раунда / итога / выбор зрителя.
- `ItemMetadata` — JSON-поле для произвольных данных: настройки баттла (`battle.sideA/sideB/rounds/finalWinner/tournament/season/stage`) и пометка про Яндекс-источник (`yandex.albumId/trackId/sourceUrl`).
- `MediaLink.kind = 'original' | 'reaction'`.
- `AllVote` — голос зрителя в плоской форме для агрегации: `{viewerId, trackPosition, roundIndex, score, winner, isBest}`.

## Голосование зрителей — поток данных

- `useItemVotes(itemId)` — хук на странице записи, загружает мои голоса + все голоса по item, держит `state.{album, tracks, battleRounds, battleFinal, bestTracks, allVotes}`. Методы `saveAlbum`, `saveTrack`, `saveBattleRound`, `saveBattleFinal`, `toggleBest` — оптимистично обновляют state и пишут в БД через Store.
- `Store.viewerVotesByItem: Map<itemId, AllVote[]>` — все голоса по всем items, грузится один раз при загрузке каталога. `applyLocalVote` обновляет мапу оптимистично после каждого save → `ScorePair` на главной видит новый голос мгновенно.
- Агрегаторы (чистые функции в `main.tsx`):
  - `aggregateTrack(allVotes, position)` — средняя по треку.
  - `aggregateAlbum(allVotes, item)` — для каждого зрителя берёт либо album-голос, либо среднюю его трековых, затем средняя по всем.
  - `aggregateBattle(allVotes, roundIndex)` — проценты A/B/draw для раунда или итога.
  - `aggregateSingleScore(allVotes)` — для одиночного трека или цифровой оценки баттла.
- `upsertViewerVote` (Store helper) — общий select+update/insert с retry на `23505` unique violation. Защита от race condition между параллельными операциями на одну строку.
- saveAll в ItemPage: `albumHandle` и `battleHandles` дёргаются sequentially (одна строка БД), `trackHandles` параллельно через `Promise.all`.

## Импорт из Яндекс.Музыки

Идея: вставляешь ссылку (альбом или конкретный трек), система сама подтягивает название, артиста, год, обложку, треклист.

### Как устроено сейчас

```
Браузер пользователя (GitHub Pages, HTTPS)
  ↓ кнопка «Импортировать» в редакторе
fetch(`${VITE_YANDEX_PROXY_URL}/yandex-music/import?url=...`)
  ↓ HTTPS-запрос на VPS
Caddy на 195.208.3.209 (HTTPS-фронт с Let's Encrypt)
  ↓ reverse_proxy на 127.0.0.1:3001
Node.js yandex-proxy (~110 строк, server/yandex-proxy/index.mjs)
  ↓ HTTP-запрос на api.music.yandex.net/albums/{id}/with-tracks
Яндекс.Музыка JSON API (видит русский IP — отдаёт 200 OK)
```

В ответе приходит готовый `YandexImportResult` с полями: `albumId`, `trackId`, `title`, `artist`, `year`, `genre`, `coverUrl`, `tracks[]`, `sourceUrl`. Дальше фронт ([src/main.tsx:879-925](../src/main.tsx#L879-L925)) заполняет форму редактора, ставит обложку, выбирает тип (`track` если 1 трек, `album` если больше).

### Почему именно так

Яндекс.Музыка **геоблокирует** свой API для не-российских IP — `api.music.yandex.net` отдаёт HTTP 451 любым серверам в EU/US. Из-за этого не работают:

- Supabase Edge Functions (Ireland) — пробовали 2026-05-18, провалились.
- Vercel / Cloudflare Workers — та же история.
- Прямой запрос из браузера — блокируется CORS (Яндекс не разрешает кросс-доменные).

Решение: маленький Node.js-прокси на VPS **в РФ** (тот же сервер где живёт Telegram-бот Кирилла `bot-napominalka`). Подробности по поддержке — в [server/yandex-proxy/README.md](../server/yandex-proxy/README.md).

### Локальная разработка

В `.env.local` указан `VITE_YANDEX_PROXY_URL=https://195.208.3.209.sslip.io` — даже в `npm run dev` импорт идёт на тот же VPS-прокси что и прод. Никаких отдельных dev-плагинов больше нет — `vite.config.ts` минимальный.

## Импорт баттлов с YouTube

Аналогичная идея для типа «баттл»: вставляешь ссылку на ролик с YouTube — заполняется название, обложка, стороны A и B (если эвристика разобрала), ссылка на оригинал.

### Архитектура

```
Браузер пользователя
  ↓ fetch('https://www.youtube.com/oembed?url=...&format=json')
YouTube oEmbed (публичный, отдаёт CORS для нашего домена)
  ↓
ответ: { title, author_name, author_url, thumbnail_url, html }
  ↓
parseBattleTitle(title, author) — эвристика в src/main.tsx
  ↓ заполняет форму редактора баттла
```

VPS-прокси для этого **не используется** — YouTube oEmbed отдаёт CORS-заголовки для GitHub Pages / кастомного домена, ходим напрямую. Это даже надёжнее: тот же VPS блокирует трафик к YouTube (как и к Telegram), а из браузера Кирилла канал в YouTube открыт нормально.

### Эвристика парсинга

Парсер `parseBattleTitle` в [src/main.tsx](../src/main.tsx) различает три формата:

1. **`standard`** — обычный баттл 1v1 или 2v2. Признак: есть ` vs ` / ` VS ` / ` против `. Пример: `ХХОС vs SHALA | КУБОК МЦ: BACK TO BASICS`.
2. **`deathmatch`** — N участников все против всех. Признак: автор канала «БОЛЬШЕ ЧЕМ БАТТЛ» или слово DEATHMATCH/ДЕЗМАТЧ в названии. Пример: `БОЛЬШЕ ЧЕМ БАТТЛ | T!MMI x Hidji 17 x AVENUE SS х HPPD х АГЕНТ ДАН`.
3. **`unknown`** — не распознали. Возвращаем сырьё, R1Fmabes дочищает руками.

Граничные случаи обкатаны: `RAM aka ГРЯЗНЫЙ РАМИРЕС` (схлопывается в одного), `x`/`х` (латинская и кириллическая), `(Versus: Fresh Blood)` (Versus-формат со скобками), хвостовой год `| 2018`.

### Чего oEmbed не даёт

Дату публикации видео. Если когда-нибудь захотим автоподстановку года — добавим YouTube Data API v3 (нужен API key из Google Cloud). См. [BACKLOG.md](BACKLOG.md).

## Что важно помнить (gotchas)

- **`base: '/'`** — прод работает на кастомном домене `rifmabes.ru`, без подпути `/album-rater/`. Если что-то ломается с 404 — смотрим SPA-фоллбек.
- **Демо vs Supabase** определяется автоматически по env. Локально без `.env.local` будешь работать на `localStorage` и не заметишь. Голосование в демо отключено.
- **`src/main.tsx` — монолит на ~5000 строк.** Не торопиться разбивать на компоненты, пока продукт ещё гуляет. Vendor-чанки в `vite.config.ts` уже разнесли react/markdown/supabase/icons, главный chunk нашего кода ~160 kB — терпимо.
- **TypeScript-сборку прогонять перед каждым push** (`npx tsc --noEmit`). CI делает `tsc -b && vite build` — если TS падает, GitHub Pages тихо остаётся на старой версии без видимой ошибки. Был кейс 2026-05-27: 3 коммита подряд упали на TS2741 «Property 'isBest' is missing».
- **Зрительские голоса в одну строку БД** — `saveMyAlbumVote` и `saveMyBattleFinalVote` пишут в одну запись (`track_position = null, round_index = null`), но в разные колонки. Update не должен затирать соседнюю колонку (`update({score})`, не `update({score, winner_side: null})`).
- **saveAll последовательный** для album + battle (одна строка → unique violation при параллельном insert). Треки параллельные (свои строки).
- **Перед разбиением длинных списков уточнять UI-формат.** Если задача звучит как «разделить», «разбить», «разнести», «не путать» записи/материалы/вкладки, сначала уточнить ожидаемый паттерн: вкладки-переключатели, фильтр, отдельные страницы или секции одна под другой.
